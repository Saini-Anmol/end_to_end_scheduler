## Review session: 2026-05-21

### D1 — L17 t0 guardrail unimplemented (Critical)
`V1/setups/guardrails.py` is a 6-line stub (docstring only). The t0 assertion
(`t0_minute + longest_min_aging_path_to(SKU) <= first_curing_start_minute`) is
referenced in the docstring but never coded. Bootstrap does NOT call it.
**File:** `V1/setups/guardrails.py` (entire file stub)  
**Spec:** CLAUDE.md L17, approach-flow step 4, Section 9 finding #10

### D2 — L3/L18 building machine spill logic ignores aging-MAX condition (High)
`V1/routes/forward_scheduler.py:239-249`: for building lots, `ordered_machines` puts
`6001` first but the "best" loop picks globally minimum `(actual_end, machine_id)`.
If `6002` finishes earlier than `6001` for any reason, the code picks `6002` without
checking whether staying on `6001` would breach aging-MAX on a component. L3 says
spill happens ONLY on aging-MAX breach. The code spills on any machine-busy condition.
**File:** `V1/routes/forward_scheduler.py:239-249`  
**Spec:** CLAUDE.md L3, L18

### D3 — L21/L15/L16 deferred stubs not acknowledged as open work items (Medium)
Three utility files are empty stubs with only docstrings:
- `V1/utilities/event_heap.py` (6 lines, stub)
- `V1/utilities/lsf_tiebreak.py` (8 lines, stub)
- `V1/utilities/reservation_table.py` (6 lines, stub)
The scheduler module's own docstring (lines 26-29) does acknowledge these as deferred.
But this means L21 (event-driven dispatch), L15 (LSF tiebreak chain), and L16 (soft-
reservation expiry + released states) are not implemented. Topo+greedy order is used
instead of event-heap dispatch.
**File:** all three stub files, `forward_scheduler.py:26-29`  
**Spec:** CLAUDE.md L15, L16, L21

### D4 — Section 16 invariant not enforced at end-of-run (Medium)
`V1/setups/guardrails.py` mentions Section 16 invariant in its docstring but has no

runtime. Expired/released event types are never generated by V1 scheduler.
**File:** `V1/setups/guardrails.py:4`, `V1/routes/forward_scheduler.py:285-301`  
**Spec:** CLAUDE.md Section 16

### D5 — on_time_flag missing from schedule.csv (Medium)

**File:** `V1/reports/writer_schedule.py:22-34`

### D7 — audit_report.md missing "raws are bottomless" statement (Low)

**File:** `V1/reports/writer_audit.py`

### D8 — 'Min' aging unit false-alarm WARN (Low)
`V1/config/enums.py` AgingUnit enum uses "Minutes" not "Min". The audit fires
AGING_UNKNOWN_UNIT for 'Min' values in the Aging Master. `unit_conversion.py` handles
'Min' correctly via `.lower()` lookup. This is a cosmetic false-alarm Warn for 18 items
(none are pilot items). Low priority but should be fixed by adding 'min'/'Min'/'HR'/'HRS'
aliases to the AgingUnit enum or expanding `_KNOWN_AGING_UNITS`.
**File:** `V1/config/enums.py:14-16`, `V1/routes/audit.py:37`  
**Spec:** None critical; cosmetic inconsistency

### NOT a defect — FEFO at_min uses earliest-possible-start (confirmed)
`V1/routes/forward_scheduler.py:163-174`: candidate_min is computed before machine
free_from is known. This is conservative and cannot produce MIN violations (confirmed
by passing diagnostics test). Using machine_free_from in FEFO would be more accurate
but the design is intentional for V1 simplicity.

### NOT a defect — Diamond BOM serves_blocks (confirmed safe)
Diamond BOM items (CPJ1218, EHT1000 each have 2 parents) do NOT cause duplicate
block_ids in serves_blocks. `explode_block()` uses a dict so each item_code appears
once per block with accumulated qty. Verified by runtime check 2026-05-21.

## Review session: 2026-05-25 (focused: MPQ + UOM + Parallel/Sequential)

### D9 — Producer lot double-consumption by multiple GT lots (Critical / by design)
`V1/routes/forward_scheduler.py:322-329, 466-481`: ingredient lots that serve N curing
blocks (e.g. BD_12843443_4__60__0001 serving b01-b07) are selected by FEFO for EACH
of the GT lots they overlap. 78 distinct producer lots are referenced by >1 GT lot.
This is correct behaviour — the ingredient lot's qty covers all served blocks — but
the reservation_log shows `created+consumed` repeated N times for the same producer,
which is misleading (looks like N-fold over-consumption). No qty accounting error
exists in V1 because qty traceability is per-block, but it should be flagged as
a known log artefact until L16 exclusive reservation is fully implemented.
**File:** `V1/routes/forward_scheduler.py:322-329`
**Not a scheduling error for V1**; becomes a correctness issue when L16 is implemented.

### D10 — bom_graph.svg and gantt_*.html are non-deterministic (Medium)
`V1/reports/writer_bom_graph.py:25`: `nx.topological_sort()` is documented as
"nonunique permutation" — not deterministic. Different runs produce different edge
layouts. Gantt HTML divs use plotly-internal `uuid.uuid4()` — different run IDs.
Only the xlsx data sheets and dag.json are byte-identical across runs.
**Files:** `V1/reports/writer_bom_graph.py:25`, `V1/reports/writer_gantt.py:63`
**Spec:** CLAUDE.md L4.6, Section 13

### D11 — 56 lot-sizing under-min warnings silently lost (Medium)
`V1/routes/lot_sizing.py` `_enforce_min_qty_or_halt()` appends warnings to
`LotsResult.warnings`. These are printed to stdout in bootstrap but are NOT
written to `audit_report.md` or any output artefact. The planner cannot see them
after the run. Items: B815, HS106, JT444, MB1232, MB231, MS51, MS52, MS53, MT270.
**File:** `V1/reports/writer_audit.py` (missing lot_sizing warnings section)

### D12 — CPJ1218 bom_output_qty ambiguous (Open Question)
CPJ1218 has two BOM rows as Output: row11 output_qty=1860 MM (feeds 162-variant cut),
row16 output_qty=1950 MM (feeds 154-variant cut). `build_graph` takes the first row
(1860). MPQ NOS→MM conversion uses 1860 for all CPJ1218 lots regardless of which cut.
This is a data ambiguity in the source BOM, not a code defect. Raise with planner.
**File:** `V1/utilities/bom_walker.py:265`

### CONFIRMED UNCHANGED — D2 Building machine spill (High, session 2026-05-21)
Still present. 17 lots on 6001, 12 on 6002, 8 on 6003. Spill is triggered by
earliest-end not by aging-MAX breach per L3.
**File:** `V1/routes/forward_scheduler.py:415-431`

### NOT FIXED — Dead production unreported (was Medium Issue 7)

these are dead production never consumed by any committed building lot.
kpi.csv has no orphaned_lots metric. No check in demand_explosion or lot_sizing
prevents sizing upstream lots for blocks whose building lot will be infeasible.
This is a systemic issue: the engine should either (a) not size upstream lots for
infeasible building blocks OR (b) report orphaned lots in kpi.csv.
Example dead lots: EHT1000__40__0015, CPJ1218__50__0015 (serve only b15).
**Residual defect: 21 dead production lots, none flagged in KPI or infeasibilities.**
