## Review session: 2026-05-21

### D1 — L17 t0 guardrail unimplemented (Critical)
`V1/setups/guardrails.py` is a 6-line stub (docstring only). The t0 assertion
(`t0_minute + longest_min_aging_path_to(SKU) <= first_curing_start_minute`) is
referenced in the docstring but never coded. Bootstrap does NOT call it.
**File:** `V1/setups/guardrails.py` (entire file stub)  
**Spec:** CLAUDE.md L17, approach-flow step 4, Section 9 finding #10

### D2 — L3/L18 building machine spill logic ignores aging-MAX condition (High)
`V1/routes/forward_scheduler.py:239-249`: for building lots, `ordered_machines` puts
`6001` first but the "best" loop picks globally minimum `(actual_end, machine_id)`.
If `6002` finishes earlier than `6001` for any reason, the code picks `6002` without
checking whether staying on `6001` would breach aging-MAX on a component. L3 says
spill happens ONLY on aging-MAX breach. The code spills on any machine-busy condition.
**File:** `V1/routes/forward_scheduler.py:239-249`  
**Spec:** CLAUDE.md L3, L18

### D3 — L21/L15/L16 deferred stubs not acknowledged as open work items (Medium)
Three utility files are empty stubs with only docstrings:
- `V1/utilities/event_heap.py` (6 lines, stub)
- `V1/utilities/lsf_tiebreak.py` (8 lines, stub)
- `V1/utilities/reservation_table.py` (6 lines, stub)
The scheduler module's own docstring (lines 26-29) does acknowledge these as deferred.
But this means L21 (event-driven dispatch), L15 (LSF tiebreak chain), and L16 (soft-
reservation expiry + released states) are not implemented. Topo+greedy order is used
instead of event-heap dispatch.
**File:** all three stub files, `forward_scheduler.py:26-29`  
**Spec:** CLAUDE.md L15, L16, L21

### D4 — Section 16 invariant not enforced at end-of-run (Medium)
`V1/setups/guardrails.py` mentions Section 16 invariant in its docstring but has no

runtime. Expired/released event types are never generated by V1 scheduler.
**File:** `V1/setups/guardrails.py:4`, `V1/routes/forward_scheduler.py:285-301`  
**Spec:** CLAUDE.md Section 16

### D8 — 'Min' aging unit false-alarm WARN (Low)
`V1/config/enums.py` AgingUnit enum uses "Minutes" not "Min". The audit fires
AGING_UNKNOWN_UNIT for 'Min' values in the Aging Master. `unit_conversion.py` handles
'Min' correctly via `.lower()` lookup. This is a cosmetic false-alarm Warn for 18 items
(none are pilot items). Low priority but should be fixed by adding 'min'/'Min'/'HR'/'HRS'
aliases to the AgingUnit enum or expanding `_KNOWN_AGING_UNITS`.
**File:** `V1/config/enums.py:14-16`, `V1/routes/audit.py:37`  
**Spec:** None critical; cosmetic inconsistency

### NOT a defect — FEFO at_min uses earliest-possible-start (confirmed)
`V1/routes/forward_scheduler.py:163-174`: candidate_min is computed before machine
free_from is known. This is conservative and cannot produce MIN violations (confirmed
by passing diagnostics test). Using machine_free_from in FEFO would be more accurate
but the design is intentional for V1 simplicity.

### NOT a defect — Diamond BOM serves_blocks (confirmed safe)
Diamond BOM items (CPJ1218, EHT1000 each have 2 parents) do NOT cause duplicate
block_ids in serves_blocks. `explode_block()` uses a dict so each item_code appears
once per block with accumulated qty. Verified by runtime check 2026-05-21.
