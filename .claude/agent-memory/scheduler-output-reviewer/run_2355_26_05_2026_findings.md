- All 23 infeasibilities show `binding_constraint = BLOCK_OVERLAP` with `share_reserved = 0.0000`.
- GT b01 is the ONLY genuinely infeasible block (IL min-aging forces GT start >= 1037 but curing deadline requires start <= 988, window = [-49, 0]).
- GT b02–b10 (and others) are wrongly infeasible. Their feasibility windows are wide (e.g., b02: [1037, 1468], b09: [4588, 4828]).
- The defect: in the multi-producer FEFO loop at `forward_scheduler.py:430-476`, the `candidate_min` is computed as `max(earliest_start, min(c.end_min + max(min_aging, transfer) for c in overlapping))`. For the FIRST ingredient (IL/SSW), earliest_start=0, so candidate_min = producer_end + min_aging. BUT after the first ingredient sets earliest_start to, e.g., 4588 (from IL min-aging), the SECOND ingredient (SSW, end=637) has candidate_min = max(4588, 757) = 4588. At at_min=4588, SSW lots ending at 637 ARE eligible (637+4320=4957 >= 4588). Yet share_reserved=0. The exact failure path needs a debugger — suspects: bom.children() key mismatch, or fefo_candidates not including early-ending lots at the time of scan.
- 10 orphaned IL lots (0001-0005, 0007-0011) produced but never consumed.
- 2 TRD infeasibilities (0031, 0032) also BLOCK_OVERLAP on T4004 — same pattern as GT.

**Concern B — 4 aging-MAX violations from lot-sizing alignment:**
- All 4 follow the same pattern: one producer lot serves multiple consumers across a span wider than max_aging.
- XMB349__20__0002 (end=3470) serves B350 lots at 3650 and 10949. Gap to 2nd=7479 > max=4320.
- B460/MB1232 and MB231/MB1232 and B355/IL follow the identical pattern.
- JIT pass targets earliest consumer, which keeps producer early; latest consumer then breaches max_aging.
- Root cause is in lot_sizing.py `_aggregate()`: the forward-aggregate loop must enforce `lot_end + aging_MAX >= latest_consumer_start` but allows a single lot to span too many blocks for non-GT items.
- **Verdict: V1 defect (not V2 scope).** CLAUDE.md Section 8.C explicitly requires this constraint.
